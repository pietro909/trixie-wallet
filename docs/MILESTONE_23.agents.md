# Milestone 23: Thumb-Reachable Wallet Actions

**Status:** Planned (promoted from Issue 1)

## Goal
Move the Wallet home screen's Send and Receive actions into the natural
one-handed thumb zone while preserving the screen's premium, calm wallet
dashboard feel.

This milestone should prove:

- Send and Receive are reachable without stretching on common iOS and Android
  phone sizes.
- The actions remain immediately visible when the Wallet tab opens.
- The dock never covers activity rows, asset rows, balance details, pull to
  refresh, or the floating bottom tab bar.
- The design feels native to the app: Inter typography, lucide icons, haptics,
  subtle press feedback, safe-area awareness, and light/dark theme parity.
- The change is local to the Wallet home screen and does not alter send,
  receive, or tab navigation behavior.

## Current State

- `app/screens/WalletScreen.tsx` renders a vertical `ScrollView`.
- The balance card appears first, followed by an inline two-button action row.
- Recent Activity, Assets, and Balance breakdown follow below the inline
  action row.
- `RootTabs.tsx` uses the existing floating glass tab bar generated by
  `makeBottomTabsOptions()`: absolute positioning, 64px height, horizontal
  margins, bottom safe-area offset, blur, and press animation.
- `WalletScreen.tsx` already reserves `paddingBottom: 120`, which protects the
  scroll content from the tab bar but not from an additional persistent action
  surface.
- The current inline action row is visually clear but only ergonomic while the
  top of the screen is within reach.

## Product Rules

- **Primary actions belong near the thumb.** Send and Receive should sit above
  the tab bar, not under the balance card.
- **No content occlusion.** Scrolled content must be able to clear both the
  action dock and the tab bar.
- **No route churn.** Tapping Send still navigates to `SendEntry`; tapping
  Receive still navigates to `ReceiveSelect`.
- **Stable muscle memory.** The order and tap targets should not jump between
  normal balance states. Any adaptive emphasis must be visual only and subtle.
- **Native control feel.** Use the existing `Button` component where possible
  so haptics, loading behavior, press scale, and disabled affordances remain
  consistent.
- **The dock is a control surface, not a card feed item.** It should float as
  part of the app chrome, visually related to the tab bar, without making the
  Wallet home feel like stacked cards.
- **Accessible by default.** Tap targets must be at least 48px high, labels must
  be explicit, and the layout must survive larger text without overlapping.
- **Theme parity.** Light and dark mode should use the same hierarchy and not
  rely on brand color alone for contrast.

## Selected Direction

Add a private `WalletActionDock` component inside `WalletScreen.tsx`.

The dock is absolutely positioned at the bottom of the Wallet screen, above the
existing tab bar. It contains two equal-width actions:

- `Send` with `ArrowUpRight`
- `Receive` with `ArrowDownLeft`

The dock should feel like a sibling of the floating tab bar:

- Horizontal inset matches the wallet page padding.
- Surface uses the current theme's translucent / elevated visual language.
- Border and shadow are subtle.
- Buttons keep the app's existing rounded, haptic, animated press treatment.
- Spacing leaves a visible gap between the dock and tab bar so the controls do
  not look merged.

The inline action row under the balance card is removed. The balance card can
then lead directly into Recent Activity, making the top half of the Wallet home
more about account state and the bottom reach zone more about action.

## Implementation Plan

### 1. Layout and Safe Area

- Import `useSafeAreaInsets` in `WalletScreen.tsx`.
- Wrap the existing `ScrollView` in a root `View`. Define a static
  `styles.root = { flex: 1 }` in `StyleSheet.create` and apply the theme
  background at render time via style composition:
  `<View style={[styles.root, { backgroundColor: theme.colors.background }]}>`.
  `theme.colors` is runtime state and cannot live inside
  `StyleSheet.create`. The `flex: 1` is required: without it the root
  collapses to content height and the absolutely positioned dock anchors
  against that height instead of the tab scene, breaking the layout on
  shorter content states (empty activity, collapsed assets).
- Keep the `ScrollView` flexed/bounded inside the root (its existing
  `style={{ backgroundColor: ... }}` should become
  `style={{ flex: 1, backgroundColor: ... }}` so it fills the root). The
  `contentContainerStyle` keeps `padding` and the increased bottom padding
  for dock + tab clearance.
- Remove the inline action row currently rendered after the balance card.
  Also delete the now-unused `styles.actions` and `styles.actionBtn`
  entries from the `StyleSheet.create` block so Biome does not flag
  unused declarations.
- Add `WalletActionDock` as an absolutely positioned sibling of the
  `ScrollView` (positioned via `left`, `right`, `bottom` relative to the
  flexed root).
- Compute the dock bottom offset from the same assumptions as
  `makeBottomTabsOptions()`. Use this exact formula:

  ```
  bottomOffset = spacing[3] + Math.max(0, insets.bottom - 6) + 64 + spacing[3]
  ```

  Where each term is:
  - `spacing[3] + Math.max(0, insets.bottom - 6)` — tab bar's bottom gap
    (its distance from the screen edge, matching `makeBottomTabsOptions()`).
  - `64` — tab bar height.
  - `spacing[3]` — dock-to-tab gap so the two surfaces read as siblings,
    not merged.
- The scroll content bottom padding must clear the dock plus the tab
  stack. The dock uses `Button`, which only guarantees `minHeight: 48`
  (see `app/components/Button.tsx:128`), so its actual height grows with
  accessibility text scaling and cannot be hard-coded. Measure the
  rendered dock with `onLayout` (store `dockHeight` in component state)
  and set `contentContainerStyle.paddingBottom` to
  `bottomOffset + dockHeight + spacing[5]`. While `dockHeight` is `0` on
  the first render, fall back to `bottomOffset + 64 + spacing[5]` so the
  initial frame is not clipped — `64` matches `Button.minHeight + 2 *
  spacing[2]` (the dock container's vertical padding from §3), which is
  also the expected steady-state at default text size.

### 2. Dock Component

- Keep `WalletActionDock` local to `WalletScreen.tsx` unless a second screen
  needs the pattern later.
- Props:
  - `theme`
  - `bottomOffset`
  - `onSend`
  - `onReceive`
- Render a horizontal two-button layout with stable dimensions and
  `gap: spacing[3]`.
- Use the existing `Button` component for both actions. This milestone
  must extend `Button` to forward an accessibility label, because the
  current `ButtonProps` (`app/components/Button.tsx:21`) does not accept
  one and the internal `Pressable` (`app/components/Button.tsx:90`) is
  not labelled — the visible "Send" / "Receive" labels do not satisfy the
  explicit accessibility copy required below. Required changes to
  `Button`:
  - Add `accessibilityLabel?: string` to `ButtonProps`.
  - Destructure it in the component signature and forward it to the
    internal `Pressable` as `accessibilityLabel={accessibilityLabel}`.
  - No other call sites need to change; the prop is optional.
- Keep Send and Receive equal width. Preserve the current navigation behavior
  and avoid adding loading state in this milestone.
- Pass these accessibility labels from the dock into the two `Button`s:
  - Send: "Send bitcoin or assets"
  - Receive: "Receive bitcoin, Lightning, or assets"

### 3. Visual Polish

- Match page horizontal padding: `left: spacing[5]`, `right: spacing[5]`.
- Use a dock container with:
  - `borderRadius: radius.lg`
  - `padding: spacing[2]`
  - `borderWidth: 1`
  - `borderColor: theme.colors.border`
  - `backgroundColor: theme.colors.surfaceSubtle` — pinned to this token
    (not `card` or `surface`). It is the closest analogue to the tab
    bar's quiet-but-elevated read while staying opaque, and it sits one
    step off the page background in both modes so the dock remains
    legible without competing with the balance card.
  - `theme.shadow("popover")` for elevation.
- The dock is **opaque-with-shadow, not blurred.** Do not wrap it in a
  `BlurView`. The visual relationship to the floating tab bar is
  conveyed by shared shape, inset, and elevation language — not by
  duplicating its glass effect. This keeps the dock readable as a
  control surface distinct from the chrome behind it.
- In dark mode, the `surfaceSubtle` token (`neutral[900]`) is already a
  step lighter than the `neutral[1000]` background, so the dock will
  not read as a heavy black slab. Verify the contrast on-device.
- Verify button text and icons remain centered at larger accessibility text
  sizes.

### 4. Scroll and Interaction Checks

- Pull to refresh should still begin from the top of the scroll view.
- Recent Activity should be visible directly after the balance card.
- Empty activity, expanded Assets, and Balance breakdown should not be hidden
  under the dock.
- The dock should remain fixed while the wallet content scrolls.
- The bottom tab bar should remain fully tappable and visually separate.

### 5. Verification

- Run `pnpm check`.
- Run `pnpm test` if the implementation touches shared components or store
  behavior. This milestone should not need store changes.
- Manual verification on iOS and Android:
  - light mode
  - dark mode
  - small phone viewport
  - tall phone viewport
  - empty activity state
  - long activity / expanded assets state
  - pull to refresh
  - Send and Receive navigation

## Out of Scope

- Changing send or receive flow behavior.
- Adding a third quick action.
- Reworking the balance card.
- Reworking the bottom tab bar.
- Introducing a global floating action system.
- Persisting user preferences for dock order or prominence.
